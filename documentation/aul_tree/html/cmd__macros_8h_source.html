73 Description
74 IO Control commands are composed of four bit fields shown below.
75 o type - The type of driver or a default SYStem level command
76  that is shared by all drivers.
77 o command - The command to be done at the driver level, in other words
78  which function to vector too within the driver jump table.
79 o component physical id - The component to which this command is directed, if any.
80 o index - index passed to the component which which channel (i.e. DIO[index], AI[index])
81 
82 
83 NOTE: there are some potential conflicts we need to deal with or verify that
84 they are not conflicts in Linux. It appears that these will not be in conflict
85 because the type will be determined to be SYS_TYPE__NONE. The exceptions are:
86 
87 
88 refer to ioctls.h: TIOCLINUX = 0x541C
89 
90 
91 
92  format:
93  bits width description
94  05:00 6 index Used to provide channel information
95  0 <= channel <= 63.
96  07:06 2 instance The instance of the device or component or module.
97  No more than 4 instances of a component or device.
98  15:08 8 type Magic number used to ID the driver commands basically.
99  We will use this as a mechanism to filter out
100  undesired commands in/out of the driver and application.
101  So long as it is placed in the correct location, it ought
102  be compatible with Linux ioctl.h
103  31:16 8 command Same as ordinal number in Linux ioctl.h, except in
104  a different bit string location.
105  31:24 8 base_id Identifies which module, device or component in which
106  the command is intended to be delivered too.
107 
108 Note: physical_id or logical_id is the combination of base_id and instance. This excludes
109 all other information within the command.

343  layer) or driver (kernel layer). Thus, if a command is sent to
